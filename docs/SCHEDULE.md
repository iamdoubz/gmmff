# Schedule — server-side encrypted transfers

Schedule transfers let you upload an encrypted file to the server and share a
download link. No simultaneous connection between sender and recipient is
required — the recipient can download hours or days later.

**The server never sees plaintext.** Files are encrypted in the browser (or CLI)
using AES-256-GCM before a single byte leaves your device. The decryption key
lives only in the URL fragment (`#key=…`) which browsers never transmit to the
server and which never appears in server access logs.

---

## Enabling Schedule

Schedule is **disabled by default**. Add to your env file:

```bash
GMMFF_SHOW_SCHEDULE=true
GMMFF_SCHEDULE_DIR=./data/schedule
```

Restart `gmmff serve`. The Schedule tab will appear in the browser UI and the
`/api/schedule/*` endpoints will become active.

---

## Security model

### Encryption

Each file is encrypted with a unique AES-256-GCM key generated by the uploader.
The key is never sent to the server. The upload protocol:

1. Uploader generates a random 256-bit key via `crypto.getRandomValues` (browser) or `crypto/rand` (CLI)
2. The filename is encrypted separately with the same key and a random nonce
3. The file is split into 256 KB chunks; each chunk is encrypted with a unique nonce:
   ```
   nonce = [4-byte big-endian chunk index] || [8 random bytes fixed per upload]
   on-disk chunk = nonce (12 B) || AES-256-GCM ciphertext || GCM auth tag (16 B)
   ```
4. A SHA-256 hash of the full ciphertext is computed and sent to the server for integrity verification
5. The server stores only ciphertext — it cannot decrypt anything without the key

### Key distribution

The key is encoded as hex and placed in the URL fragment:

```
https://host/?type=schedule&id={file_id}#key={hex_key}
```

URL fragments (`#...`) are a browser-only concept — they are never transmitted
in HTTP requests and never appear in server access logs, nginx access logs, or
any proxy logs. The server operator cannot recover the key from logs.

**Risk:** if the recipient shares the full URL in a plaintext channel (e.g. email,
Slack), the key is visible in that channel. This is an inherent property of
fragment-based key sharing. For higher security, distribute the key and the
`?id=` part via separate channels.

### Delete key

A short unique delete key is generated at upload time and shown only to the
uploader. It is stored as-is in the server's `.meta` sidecar. The delete
endpoint requires both the `file_id` and the `delete_key` — possession of
either alone is insufficient.

## Access control

### Upload access

Controlled by two env vars that work together:

| Scenario | Result |
|----------|--------|
| Neither `UPLOAD_IP` nor `PASSWORD` set | Anyone can upload |
| `UPLOAD_IP` set, caller's IP matches | Upload allowed, no password |
| `UPLOAD_IP` set, caller's IP doesn't match, `PASSWORD` set | Password required |
| `UPLOAD_IP` set, caller's IP doesn't match, no `PASSWORD` | Upload blocked |
| Only `PASSWORD` set | Password required for everyone |

The browser UI gates the entire Create workflow — the password prompt appears
before the file picker. The CLI prompts interactively if no `--password` flag
is provided.

### Download access

`GMMFF_SCHEDULE_DOWNLOAD_IP` controls who can download. Default `0.0.0.0`
means anyone with the link can download. Set to a CIDR/IP list to restrict.

## Nginx configuration

Two location blocks are required in addition to the standard gmmff config.
See `configs/gmmff.conf` for the complete example.

```nginx
# Auth — needs real client IP for access control
location = /api/schedule/auth {
    proxy_pass         http://gmmff_backend;
    proxy_http_version 1.1;
    proxy_set_header   Host             $host;
    proxy_set_header   X-Real-IP        $remote_addr;
    proxy_set_header   X-Forwarded-For  $proxy_add_x_forwarded_for;
    proxy_read_timeout 10s;
    proxy_send_timeout 10s;
}

# Uploads — needs IP + large body limit + long timeouts
location /api/schedule/upload {
    proxy_pass         http://gmmff_backend;
    proxy_http_version 1.1;
    proxy_set_header   Host             $host;
    proxy_set_header   X-Real-IP        $remote_addr;
    proxy_set_header   X-Forwarded-For  $proxy_add_x_forwarded_for;
    proxy_buffering    off;
    proxy_read_timeout 1200s;
    proxy_send_timeout 1200s;
    client_max_body_size 1025M;  # must exceed GMMFF_SCHEDULE_MAX_SIZE
}
```

## Environment variables

| Env var | Default | Description |
|---------|---------|-------------|
| `GMMFF_SHOW_SCHEDULE` | `false` | Enable the Schedule tab and API endpoints |
| `GMMFF_SCHEDULE_DIR` | `./data/schedule` | Storage root; `pending/` and `complete/` are created automatically |
| `GMMFF_SCHEDULE_MAX_SIZE` | `1gb` | Maximum upload size; supports `gb`/`mb`/`kb` suffix |
| `GMMFF_SCHEDULE_MAX_DOWNLOADS` | `1` | Server-wide cap on downloads per file; `0` = unlimited |
| `GMMFF_SCHEDULE_UPLOAD_IP` | — | Comma-separated IPs/CIDRs allowed to upload without a password |
| `GMMFF_SCHEDULE_PASSWORD` | — | Required upload password (bypassed if caller's IP is in `UPLOAD_IP`) |
| `GMMFF_SCHEDULE_DOWNLOAD_IP` | `0.0.0.0` | Comma-separated IPs/CIDRs allowed to download; `0.0.0.0` = anyone |
| `GMMFF_SCHEDULE_CLEANUP_INTERVAL` | — | Crontab-format background cleanup, e.g. `*/30 * * * *` |
| `GMMFF_TTL_SETTINGS` | `1h,8h,1 day,3 days,7 days,30 days` | Comma-separated TTL options for the upload dropdown |

### TTL format

`GMMFF_TTL_SETTINGS` accepts flexible formats — all of these are valid:

```bash
GMMFF_TTL_SETTINGS=1h,8h,1 day,3 days,7 days,30 days
GMMFF_TTL_SETTINGS=30m,2h,12h,1d,3d,7d
GMMFF_TTL_SETTINGS=1 hour,1 day,1 week,1 month
```

## Storage layout

```
$GMMFF_SCHEDULE_DIR/
  pending/
    {upload_id}.enc     ← ciphertext chunks appended as they arrive
    {upload_id}.meta    ← JSON: chunks received, total, expires_at, chunk_size
  complete/
    {file_id}.enc       ← finalized ciphertext (moved from pending/ on complete)
    {file_id}.meta      ← JSON: filename_enc, sha256_cipher, expires_at, delete_key, downloads_left
```

Keeping pending and complete in separate directories is intentional — on
Docker deployments you can mount `complete/` on persistent storage (e.g. an
NFS volume or a dedicated SSD) while keeping `pending/` on a fast tmpfs,
reducing wear on the persistent storage from incomplete writes.

## Cleanup

Two modes:

**Background goroutine** — runs inside `gmmff serve`, no external process needed:

```bash
GMMFF_SCHEDULE_CLEANUP_INTERVAL=*/30 * * * *
```

**One-shot via cron** — runs `gmmff cleanup`, exits immediately:

```bash
# /etc/cron.d/gmmff-cleanup
*/30 * * * *  gmmff  /usr/local/bin/gmmff cleanup
```

Both modes remove the same things:
- Completed files past their `expires_at` timestamp
- Completed files where `downloads_left == 0`
- Pending uploads created more than 24 hours ago (assumed abandoned)

## CLI usage

See [CMDS.md](CMDS.md) for the full flag reference.

### Upload

```bash
# Single file, 24h TTL (default)
gmmff schedule upload report.pdf --server https://your-server

# Multiple files — zipped automatically
gmmff schedule upload notes.txt data.csv --ttl 8h

# Directory
gmmff schedule upload ./project/ --ttl 7d --max-downloads 3

# With QR code in terminal
gmmff schedule upload file.zip --qr

# JSON output (for scripting)
gmmff schedule upload file.zip --json

# Save share URL to a file
gmmff schedule upload file.zip --out share.txt
```

Example output:

```
Uploading report.pdf (2.4 MB)...

  Upload complete!

  File ID:         a1b2c3d4e5f6...
  Decryption key:  deadbeef1234...

  Share URL:       https://your-server/?type=schedule&id=a1b2c3d4e5f6
  Full URL:        https://your-server/?type=schedule&id=a1b2c3d4e5f6#key=deadbeef1234...

  Auto-download:   https://your-server/?type=schedule&id=a1b2c3d4e5f6&dl=1#key=deadbeef1234...

  Delete URL:      https://your-server/?type=schedule&id=a1b2c3d4e5f6&action=delete&dk=ff001122

  Expires:         2026-05-22 09:00 CDT (in 24 hours)
  Avg speed:       3.2 MB/s
```

### Download

```bash
# Download to current directory (filename from server)
gmmff schedule download "https://your-server/?type=schedule&id=X#key=Y"

# Download to a specific directory
gmmff schedule download "https://your-server/...#key=..." --out ./downloads/

# Stream to stdout (for piping)
gmmff schedule download "https://your-server/...#key=..." --out - > file.zip
gmmff schedule download "https://your-server/...#key=..." --out - | tar xz

# Prompt before consuming the last download
gmmff schedule download "https://your-server/...#key=..." --confirm

# JSON output
gmmff schedule download "https://your-server/...#key=..." --json
```

**Note on shell quoting:** The `#` in the URL is a comment character in most
shells. Always quote the URL with double quotes.

### Delete

```bash
# From the delete URL shown after upload
gmmff schedule delete "https://your-server/?type=schedule&id=X&action=delete&dk=Y"

# Explicit flags
gmmff schedule delete --id abc123 --delete-key ff001122 --server https://your-server
```

## Browser ↔ CLI interoperability

A file uploaded via the browser can be downloaded via the CLI and vice versa.
Both use the identical wire format:

| Parameter | Value |
|-----------|-------|
| Encryption | AES-256-GCM, 256-bit key |
| Chunk size | 256 KB plaintext (configurable via `--chunk-size`) |
| Nonce | `[4-byte big-endian chunk index][8 random bytes]` |
| On-disk layout | `nonce (12 B) \|\| ciphertext \|\| GCM tag (16 B)` |
| Filename | AES-256-GCM encrypted, hex-encoded in `.meta` sidecar |
| Integrity | SHA-256 of full ciphertext, verified at upload complete |
| Key encoding | Lowercase hex in URL fragment `#key=` |

## API reference

All endpoints are under `/api/schedule/`. They return JSON on error:
`{ "error": "message" }`.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/api/schedule/auth` | — | Check upload access status for caller's IP |
| `POST` | `/api/schedule/upload/init` | IP/password | Create a pending upload slot |
| `POST` | `/api/schedule/upload/chunk` | — | Append one encrypted chunk |
| `POST` | `/api/schedule/upload/complete` | — | Finalize upload, get `file_id` |
| `GET` | `/api/schedule/download/{fileID}` | Download IP | Stream ciphertext |
| `GET` | `/api/schedule/meta/{fileID}` | Download IP | Public metadata (no delete key) |
| `DELETE` | `/api/schedule/delete/{fileID}/{deleteKey}` | — | Delete file |
| `GET` | `/api/schedule/ttl-options` | — | List configured TTL options |

### Share URL format

```
# Standard share URL
https://host/?type=schedule&id={file_id}#key={hex_key}

# Auto-download (browser starts download on page load, or for wget/curl scripts)
https://host/?type=schedule&id={file_id}&dl=1#key={hex_key}

# Delete URL (uploader only)
https://host/?type=schedule&id={file_id}&action=delete&dk={delete_key}
```
